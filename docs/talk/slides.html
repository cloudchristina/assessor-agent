IRAP and ISM — what they are, why they matter

Australia's framework for handling government and regulated data — and the bill that comes with it

IRAP

Information Security Registered Assessors Program

Run by the Australian Signals Directorate (ASD). Independent assessors verify that a system meets ISM controls before it can handle OFFICIAL, PROTECTED, or SECRET data.

Required for: any cloud / SaaS handling AU government workloads. Banks rely on equivalent rigour for APRA CPS 234.

ISM

Information Security Manual · ASD

1,000+ controls across six functions: govern · identify · protect · detect · respond · recover. Updated quarterly — latest March 2026.

Today's talk hits ISM-1546, 1509, 1555, 1175, 0445, 1545 — the access-control family.

Stage I

Deterministic foundation — six rules, six Lambdas

Why Lambda?

  • Per-Lambda IAM · least-privilege is automatic. extract-uar can read SQL Server; rules-engine can't.
  • Per-Lambda OTel span · every step is one segment in X-Ray. Auditor's trace is free.
  • Cold-start ok · weekly + monthly cadence; we accept 800ms to save the orchestration overhead.
  • Pay-per-invocation · ~$0.30 / month for all 10 Lambdas at this volume.

Challenges we hit

01 · Rules correctness

Six rules → six per-rule golden suites + counterfactual tests. Flipping one input must flip only the expected rule.

02 · Provable coverage

Manifest SHA-256 of sorted row-IDs. Auditor re-derives it. "We didn't miss a row" becomes a hash check, not a promise.

03 · Speaks auditor

Every finding carries its ISM control + structured evidence dict. The finding answers "why?" inline.

Guardrails, evals, and managing hallucinations

Multi-layered defence — the consensus pattern from Anthropic, AWS, and the agent-safety literature

L1
Prompt & input controls
Raw rows never reach the model · temp=0 · structured output via Bedrock tool-use · agent given "return Unknown" escape hatch (Anthropic eval guidance)
L2
Runtime guardrails
4 deterministic Lambdas: schema · citation · reconciliation · entity-grounding · <100ms · $0/run · runs after every agent call
L3
Cross-model validation
Haiku 4.5 LLM-as-judge · adversarial probe · self-consistency on CRITICAL findings · second-opinion model from a different family
L4
Offline evals (CI)
Golden set · adversarial set · counterfactual diffs · Hypothesis property tests · blocks merge on regression — in flight (Plan 2)
L5
Production drift detection
Shadow evals on 5% of prod runs · reviewer-disagreement loop · weekly KS test on finding distribution · circuit breaker — post-MVP

Coral border = deployed today. Blue border = roadmap. The live prompt-injection attempt will trip L2 citation-guardrail — you'll see it fire on stage.

Tracing — every tool call is a signed span

One env var unlocks all of this. STRANDS_TELEMETRY_ENABLED=true → ADOT layer → X-Ray.

Why this matters for compliance

  • Every tool call is a span — what was asked, what was returned, how long it took.
  • Every Bedrock invoke is a subsegment with input + output token counts.
  • Every gate has a passed=true|false attribute. Quarantined runs are obvious.
  • Every span carries the run_id. One trace ID, full forensic replay.

The wiring

env:
  STRANDS_TELEMETRY_ENABLED: true
  OTEL_EXPORTER_OTLP_TRACES_ENDPOINT: …
layers:
  - arn:…:layer:adot-python:1

Adding a tool = adding a span = adding a slice of audit trail. There is no manual instrumentation.
